## Problem Statement

PPanel 当前需要 3 个容器（server + admin-web + user-web）加上 MySQL 和 Redis 才能完整部署。对于开源面板来说，这个复杂度劝退了大量潜在用户。目标是实现「一个 Docker 就能跑」，Go server 内嵌前端静态产物，单进程、单端口、单镜像。

## What Makes This Cool

`docker run -p 8080:8080 ppanel` 一行命令启动完整的管理后台 + 用户面板 + API 服务。镜像从 ~400MB 缩减到 ~80MB。这是 Gitea/Memos/Grafana 级别的开源部署体验。

## Migration Strategy: 渐进式（三方审查共识）

采纳审查建议的 **Phase 1 → Phase 2** 渐进式迁移，而非 Big Bang：

- **Phase 1（本 PR）：仅 embed admin** — Admin 只有 2 个 SSR 依赖（layout cookies + generateMetadata），迁移风险最低，可快速验证整套 embed + SPA fallback + env 注入管线
- **Phase 2（后续 PR）：embed user** — User 有 11+ SSR 页面 + async Server Components + redirect()，需单独 PR 处理
- User 面板在 Phase 1 期间保持现有独立部署（docker-compose 中的 user-web 服务）

## Constraints

- 前端当前使用 Next.js 16 App Router + `output: "standalone"` + SSR
- i18n 使用 `next-intl` v4，通过 `getRequestConfig()` 服务端动态导入
- Root Layout 依赖 `cookies()` + `headers()` + SSR API 调用
- 无 API Routes，无 Server Actions，无 middleware
- 项目已在使用 `embed.FS`（初始化模板 + SQL 迁移）
- MySQL 和 Redis 仍为外部依赖（不在本次范围内）

## Premises

1. 两个前端都是 dashboard（管理后台 + 用户面板），SSR 对它们没有实际价值。管理后台完全在登录后访问。用户面板首页是公开落地页（Hero/Stats/ProductShowcase），静态导出后首屏为预渲染 HTML（但无动态数据），动态内容通过客户端 API 加载。这对 SEO 可接受（HTML 结构在），但 FCP 会有短暂空壳。
2. 为了「一个 Docker 就能跑」，值得把 Next.js 从 standalone 模式改为静态导出。放弃 SSR，换来的是：不需要 Node.js 运行时，Go 二进制直接供给前端，镜像从 ~300MB 减小到 ~30MB。
3. 前端嵌入 Go 后，API URL 问题自然消失。前端和后端同源，不再需要配置 NEXT_PUBLIC_API_URL，前端直接调用同域的 /v1/* API。
4. i18n 可以完全在客户端实现。当前只支持 en-US 和 zh-CN 两种语言，用户通过 cookie 切换。客户端从 cookie/navigator.language 读取 locale，动态加载对应 JSON 即可。

## Approaches Considered

### Approach A: 静态导出 + Go embed (chosen)

Next.js `output: "export"` 生成纯静态 HTML/CSS/JS，通过 `//go:embed` 嵌入 Go 二进制，Gin 的 `StaticFS` 中间件供给前端。

- Effort: M (CC+gstack: ~2-3h, 人工: ~1-2 周)
- Risk: Med
- Pros: Gitea 级别部署体验，~50-70MB 镜像，单进程单端口，无 Node.js 依赖
- Cons: 需要重构前端 SSR 逻辑
- Reuses: embed.FS 已用于初始化模板（server/initialize/config.go）

### Approach B: 单镜像多进程 + s6-overlay

一个 Alpine 镜像装 Go 二进制 + Node.js 22，用 s6-overlay 管理 3 个进程。Go 内建反向代理转发前端请求。

- Effort: S
- Risk: Low
- Pros: 零代码修改，保留 SSR，最快能发布
- Cons: 镜像 ~400MB，3 个进程调试复杂，依赖 Node.js 运行时
- Reuses: 现有 Dockerfile

## Recommended Approach

**Approach A: 静态导出 + Go embed**

开源项目的部署体验是用户的第一印象。~60MB vs ~400MB、单进程 vs 三进程的差距是决定性的。

#### 2. i18n 重构: 服务端 → 客户端

**当前**（服务端 getRequestConfig）：
```typescript
export default getRequestConfig(async () => {
  const cookieLocale = (await cookies()).get("locale")?.value;
  const messages = {
    menu: (await import(`./${locale}/menu.json`)).default,
    // ...
  };
  return { locale, messages };
});
```

**目标**（客户端 Provider）：
```typescript
// locales/client.ts
export function getClientLocale(): string {
  if (typeof document !== "undefined") {
    const cookie = document.cookie.match(/locale=([^;]+)/);
    if (cookie) return cookie[1];
  }
  return navigator?.language?.startsWith("zh") ? "zh-CN" : "en-US";
}

// app/layout.tsx — 改为 client component 或使用 NextIntlClientProvider
// 预加载所有 locale JSON（只有 2 种语言，体积可控）
```

关键决策：只有 en-US 和 zh-CN 两种语言。可以预加载全部 locale JSON（合计 ~100-200KB gzip），通过客户端 Provider 切换，无需动态导入。

**i18n 方案选择：** 保留 `next-intl` 的 `NextIntlClientProvider`（纯客户端模式），不换库。具体做法：
1. 移除 `createNextIntlPlugin` wrapper（它注入的是服务端 request config）
2. 在 Root Layout 中直接导入所有 locale JSON，传给 `NextIntlClientProvider`
3. 所有使用 `getTranslations()` 的服务端页面改为 `useTranslations()`（客户端 hook）
4. `getLocale()` / `getMessages()` 改为从客户端 Provider 获取

**Plan B:** 如果 `next-intl` 纯客户端模式在静态导出下有兼容性问题，替换为 `react-intl`（API 相似，迁移成本低）。

**需要迁移的服务端 i18n 调用：**
- `apps/user/app/oauth/[platform]/page.tsx` — `getTranslations()`
- `apps/user/app/bind/[platform]/page.tsx` — `getTranslations()`
- `apps/user/locales/request.ts` — `getRequestConfig()` (删除)
- `apps/admin/locales/request.ts` — `getRequestConfig()` (删除)

#### 3. Root Layout 重构: SSR → 客户端

**当前**（SSR 数据获取）：
```typescript
export default async function RootLayout({ children }) {
  const Authorization = (await cookies()).get("Authorization")?.value;
  const { data } = await getGlobalConfig({ ... });
  // ...
}
```

**目标**（客户端初始化）：
```typescript
"use client";
export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <Providers>      {/* React Query + Zustand + i18n */}
          <AuthGuard>     {/* 客户端检查 cookie，无 auth 则 redirect */}
            <AppInitializer> {/* useEffect 加载 globalConfig */}
              {children}
            </AppInitializer>
          </AuthGuard>
        </Providers>
      </body>
    </html>
  );
}
```

#### 4.5 `next-runtime-env` 迁移方案

静态导出后 `env()` from `next-runtime-env` 不可用。替代方案：Go server 在 serve `index.html` 时注入 `<script>window.__ENV={...}</script>`。

**影响范围：** admin 8 个环境变量 / user 18 个环境变量，合计 ~39 个组件文件引用。

**迁移步骤：**
1. Go server 新增模板注入逻辑：读取 `index.html`，在 `<head>` 中插入 `<script>window.__ENV=JSON.parse('...')</script>`
2. 环境变量从系统环境变量或配置文件读取
3. 前端所有 `env("KEY")` 替换为 `window.__ENV?.KEY ?? defaultValue`
4. 移除 `next-runtime-env` 依赖和 `<PublicEnvScript />`

**开发模式兼容：** 开发时仍用 `.env.local` + `process.env.NEXT_PUBLIC_*`，生产时由 Go 注入。

#### 5. Go Server 嵌入前端

> **审查修复 C1:** embed.FS 是编译时不可变的，不能运行时修改 index.html。解决方案：启动时读出 index.html 为 `[]byte`，用 `bytes.Replace` 注入 `window.__ENV`，缓存在内存中。
>
> **审查修复 C2:** `r.StaticFS()` 不支持 SPA fallback。Admin 和 User 都使用自定义 handler，对不存在的路径 fallback 到 index.html。

```go
// server/web/static.go (Phase 1: 仅 admin)
package web

import (
    "bytes"
    "embed"
    "encoding/json"
    "fmt"
    "io/fs"
    "net/http"
    "strings"
    "sync"

    "github.com/gin-gonic/gin"
)

//go:embed admin-dist/*
var adminFS embed.FS

var (
    adminIndexHTML []byte
    adminSub      fs.FS
    adminOnce     sync.Once
)

// InitStatic 在服务启动时调用，注入运行时环境变量到 index.html
func InitStatic(envVars map[string]string) error {
    var initErr error
    adminOnce.Do(func() {
        var err error
        adminSub, err = fs.Sub(adminFS, "admin-dist")
        if err != nil {
            initErr = fmt.Errorf("admin-dist sub fs: %w", err)
            return
        }

        raw, err := fs.ReadFile(adminFS, "admin-dist/index.html")
        if err != nil {
            initErr = fmt.Errorf("read admin index.html: %w", err)
            return
        }

        // 注入 window.__ENV（启动时一次性，非每次请求）
        envJSON, _ := json.Marshal(envVars)
        adminIndexHTML = bytes.Replace(raw,
            []byte("</head>"),
            []byte(fmt.Sprintf(`<script>window.__ENV=%s</script></head>`, envJSON)),
            1,
        )
    })
    return initErr
}

func RegisterStaticRoutes(r *gin.Engine) {
    adminHTTP := http.FS(adminSub)

    // Admin SPA: /admin/* — 自定义 handler 支持 SPA fallback
    r.GET("/admin", serveAdminIndex)
    r.GET("/admin/*filepath", func(c *gin.Context) {
        filepath := c.Param("filepath")

        // 尝试提供静态文件（JS/CSS/images）
        f, err := adminHTTP.Open(strings.TrimPrefix(filepath, "/"))
        if err == nil {
            defer f.Close()
            // 静态资源直接供给
            http.StripPrefix("/admin", http.FileServer(adminHTTP)).ServeHTTP(c.Writer, c.Request)
            return
        }

        // SPA fallback: 非静态文件路径返回 index.html（含注入的 __ENV）
        serveAdminIndex(c)
    })

    // Phase 2 TODO: User 前端 NoRoute handler
    r.NoRoute(func(c *gin.Context) {
        path := c.Request.URL.Path
        if strings.HasPrefix(path, "/v1/") {
            c.JSON(404, gin.H{"msg": "not found"})
            return
        }
        // Phase 1: user 未 embed，返回 404
        c.JSON(404, gin.H{"msg": "not found"})
    })
}

func serveAdminIndex(c *gin.Context) {
    c.Data(http.StatusOK, "text/html; charset=utf-8", adminIndexHTML)
}
```

> **关键变更说明：**
> 1. embed 目录从 `internal/handler/` 移到 `server/web/`，避免污染生成代码目录
> 2. `InitStatic()` 在启动时将 embed.FS 中的 index.html 读出并注入 `window.__ENV`，结果缓存为 `[]byte`，解决 embed.FS 不可变问题
> 3. Admin 不再使用 `r.StaticFS()`，改为 `r.GET("/admin/*filepath")` + 文件存在性检查 + SPA fallback，解决 `/admin/dashboard/users` 404 问题
> 4. Phase 1 只 embed admin，NoRoute 暂不处理 user

## Open Questions

1. **开发模式如何处理？** 开发时前端仍用 `bun run dev` + HMR，Go server 单独跑。`make dev APP=admin` 保持不变。只有生产构建才 embed。
2. **Next.js Image 组件？** 静态导出不支持 Image Optimization API。需要设置 `images.unoptimized: true`，改用原生 `<img>` 或第三方 CDN 处理图片优化。
3. **现有 `docker/web.Dockerfile` 和 `docker-compose.yml` 的去留？** 实施后保留为可选的分离部署方案（大规模部署时可能需要前后端分离扩展）。
4. **embed 目录位置？** `internal/handler/` 是生成代码目录，放几十 MB 前端产物不太协调。可考虑放在 `server/web/` 顶层包，或使用 `.gitignore` 忽略构建产物。
