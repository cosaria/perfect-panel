## Problem Statement

perfect-panel 的 server 使用 gopure（goctl 变体）从 `.api` 文件生成 handler 和 types，但 `.api` 是 go-zero 专有 DSL，生态中没有成熟的 OpenAPI 3.x 生成工具。前端 API client 依赖外部仓库 ppanel-docs 的 Swagger JSON，每次 API 变更需要三步手动操作。现有响应格式全部 HTTP 200 + body 业务码，OpenAPI spec 无法准确描述错误场景。

## What Makes This Cool

一个命令，从 Go 类型到 TypeScript SDK 全部自动更新。server 启动时自动暴露 OpenAPI 3.1 spec。前端直接用 `response.ok` 判断成功/失败，不再需要解包 `data.data?.list`。错误响应有精确的 HTTP 状态码和 RFC 9457 格式。AI 助手可以直接从 `/openapi.json` 理解所有接口。

## Constraints

- Gin 不可替换（250 个文件引用，是项目的 HTTP 框架基石）
- OpenAPI 3.1+ 是硬性要求
- 257 个 logic 文件不动（业务逻辑层完全保留）
- 现有 Gin 中间件（JWT、CORS、Device、Server）全部保持原样
- 需要支持渐进式迁移（新旧路由可共存）

## Premises

1. 抛弃 `.api` 文件和 `gopure` — handler 层从自动生成变为手写（huma v2 格式）
2. 257 个 logic 文件完全不动 — huma handler 只是调用 logic 的薄 wrapper
3. `types.go`（2805 行）保留为共享类型定义文件，不再自动生成
4. 现有 Gin 中间件全部保持原样，在 `humagin.New()` 之前注册
5. OpenAPI 3.1 spec 由 huma 运行时生成，通过 CLI 命令导出 JSON 文件供 `@hey-api/openapi-ts` 消费
6. 切换到正确 HTTP 状态码 — 去掉 `{code, msg, data}` 信封，用 huma 原生 RFC 9457 错误格式

## 一、现有逻辑（BEFORE）

### 1.1 Server 架构

```
server/apis/swagger_{admin,common,user}.api（go-zero DSL）
    │
    └──→ gopure（自定义 goctl）自动生成：
         ├── internal/handler/ — 243 个 handler 文件（15-25 行的薄 wrapper）
         ├── internal/handler/routes.go — 路由注册（DO NOT EDIT）
         └── internal/types/types.go — 2805 行请求/响应结构体（DO NOT EDIT）
              │
              └── internal/logic/ — 257 个 logic 文件（手写业务逻辑）
```

### 1.2 Handler 模式（现有）

```go
// internal/handler/admin/ads/createAdsHandler.go（生成代码）
func CreateAdsHandler(svcCtx *svc.ServiceContext) func(c *gin.Context) {
    return func(c *gin.Context) {
        var req types.CreateAdsRequest
        _ = c.ShouldBind(&req)
        validateErr := svcCtx.Validate(&req)
        if validateErr != nil {
            result.ParamErrorResult(c, validateErr)
            return
        }
        ctx := c.Request.Context()
        l := ads.NewCreateAdsLogic(ctx, svcCtx)
        err := l.CreateAds(&req)
        result.HttpResult(c, nil, err)
    }
}
```

### 1.3 响应格式（现有）

```json
// 成功 — HTTP 200
{ "code": 200, "msg": "success", "data": { "list": [...], "total": 10 } }

// 业务错误 — HTTP 200（!）
{ "code": 40002, "msg": "登录过期" }

// 系统错误 — HTTP 200（!）
{ "code": 50001, "msg": "internal error" }
```

所有响应都是 HTTP 200。错误通过 body 中的 `code` 字段区分。前端拦截器需要解析 body 并手动判断：

```typescript
// apps/admin/utils/request.ts
requset.interceptors.response.use(async (response) => {
    const { code } = response.data;
    if (code !== 200) {
        await handleError(response);  // 40002-40005 登出，其余 toast
        throw response;
    }
    return response;
});
```

### 1.4 前端调用模式（现有）

```typescript
// data.data 双层解包
const { data } = useQuery({
    queryKey: ["getPaymentPlatform"],
    queryFn: async () => {
        const { data } = await getPaymentPlatform();
        return data?.data?.list || [];   // axios 的 data + 业务信封的 data
    },
});
```

### 1.5 现有问题汇总

| 问题 | 影响 |
|------|------|
| `.api` 是 go-zero 专有 DSL | 无成熟的 OpenAPI 3.1 生成工具 |
| 依赖外部 ppanel-docs 仓库 | API 变更无法端到端自动化 |
| 所有响应 HTTP 200 | OpenAPI 无法准确描述错误场景 |
| `data.data?.list` 双层解包 | 开发体验差，容易出错 |
| 手动 queryKey 字符串 | 缓存失效 bug 风险 |
| axios 依赖 | 额外 bundle size |

## 二、重构后逻辑（AFTER）

### 2.1 Server 架构

```
internal/types/types.go（手动维护，共享类型定义）
    │
    ├── internal/handler/ — 243 个 huma handler（手写，~10 行/个）
    │   └── 每个 handler 定义 Input/Output 类型 + 调用 logic
    │
    ├── internal/handler/routes.go — huma 路由注册（手写）
    │   └── humagin.NewWithGroup() 适配 Gin + 注册所有 huma.Register()
    │
    └── internal/logic/ — 257 个 logic 文件（不动）

运行时：
    Gin + huma v2 (humagin adapter)
        │
        ├──→ /openapi.json（自动暴露 OpenAPI 3.1 spec）
        ├──→ /docs（Swagger UI / Scalar）
        │
        └──→ CLI: go run . openapi export → docs/openapi/{admin,common,user}.json
             │
             ├──→ @hey-api/openapi-ts → TypeScript SDK + TanStack Query hooks
             └──→ 提交到 git，AI 可直接读取
```

### 2.2 Handler 模式（重构后）

```go
// internal/handler/admin/ads/createAdsHandler.go（手写，huma v2 格式）
package ads

import (
    "context"
    "github.com/perfect-panel/server/internal/logic/admin/ads"
    "github.com/perfect-panel/server/internal/svc"
    "github.com/perfect-panel/server/internal/types"
)

type CreateAdsInput struct {
    Body types.CreateAdsRequest
}

type CreateAdsOutput struct {
    Body struct{} // 无返回数据时用空 struct；有数据时用具体类型如 *types.AdsData
}

func CreateAdsHandler(svcCtx *svc.ServiceContext) func(context.Context, *CreateAdsInput) (*CreateAdsOutput, error) {
    return func(ctx context.Context, input *CreateAdsInput) (*CreateAdsOutput, error) {
        l := ads.NewCreateAdsLogic(ctx, svcCtx)
        err := l.CreateAds(&input.Body)
        if err != nil {
            return nil, err // huma 自动转为适当的 HTTP 错误
        }
        return &CreateAdsOutput{}, nil
    }
}
```

### 2.3 路由注册（重构后）

```go
// internal/handler/routes.go（手写，替代生成的 routes.go）
package handler

import (
    "net/http"
    "github.com/danielgtaylor/huma/v2"
    "github.com/danielgtaylor/huma/v2/adapters/humagin"
    "github.com/gin-gonic/gin"
    adminAds "github.com/perfect-panel/server/internal/handler/admin/ads"
    "github.com/perfect-panel/server/internal/middleware"
    "github.com/perfect-panel/server/internal/svc"
)

func RegisterHandlers(router *gin.Engine, svcCtx *svc.ServiceContext) {
    // 1. 现有 Gin 中间件不动
    adminGroup := router.Group("/v1/admin")
    adminGroup.Use(middleware.AuthMiddleware(svcCtx))

    // 2. 用 humagin 包装路由组
    config := huma.DefaultConfig("Perfect Panel Admin API", "1.0.0")
    config.Components.SecuritySchemes = map[string]*huma.SecurityScheme{
        "bearer": {Type: "http", Scheme: "bearer", BearerFormat: "JWT"},
    }
    api := humagin.NewWithGroup(router, adminGroup, config)

    // 3. 注册 huma 路由
    huma.Register(api, huma.Operation{
        OperationID: "createAds",
        Method:      http.MethodPost,
        Path:        "/v1/admin/ads",
        Summary:     "Create advertisement",
        Tags:        []string{"ads"},
        Security:    []map[string][]string{{"bearer": {}}},
    }, adminAds.CreateAdsHandler(svcCtx))

    // ... 其余 242 个路由
}
```

### 2.5 错误码映射

现有业务码 → HTTP 状态码：

| 现有 code | 含义 | → HTTP Status |
|-----------|------|---------------|
| 200 | 成功 | 200 OK |
| 40002 | 未登录 | 401 Unauthorized |
| 40003 | Token 过期 | 401 Unauthorized |
| 40004 | 权限不足 | 403 Forbidden |
| 40005 | 设备限制 | 403 Forbidden |
| 10xxx | 数据库错误 | 500 Internal Server Error |
| 20xxx | 用户操作错误 | 400 Bad Request 或 422 |
| 30xxx | 节点错误 | 502 Bad Gateway |

`pkg/xerr/` 中的错误码需要统一映射到 HTTP 状态码。**采用单一方案：在 `pkg/xerr/` 中增加 `StatusCode() int` 方法**，让每个 xerr 错误实现 `huma.StatusError` 接口。huma 检测到返回的 error 实现了 `GetStatus() int` 时，自动使用对应的 HTTP 状态码：

```go
// pkg/xerr/errors.go — 增加方法
func (e *CodeError) GetStatus() int {
    switch {
    case e.Code >= 40002 && e.Code <= 40005:
        return http.StatusUnauthorized // 401
    case e.Code >= 20000 && e.Code < 30000:
        return http.StatusBadRequest // 400
    case e.Code >= 30000 && e.Code < 40000:
        return http.StatusBadGateway // 502
    default:
        return http.StatusInternalServerError // 500
    }
}

func (e *CodeError) Error() string { return e.Msg }
```

这样 logic 层不需要任何改动，现有的 `return xerr.NewCodeError(...)` 自动产生正确的 HTTP 状态码。

### 2.6 前端拦截器（重构后）

```typescript
// apps/admin/utils/setup-clients.ts
import { client } from '@/services/admin/client.gen';

// 请求拦截器：JWT 注入
client.interceptors.request.use(async (request) => {
    const token = getAuthorization();
    if (token) request.headers.set('Authorization', token);
    return request;
});

// 响应拦截器：大幅简化
client.interceptors.response.use(async (response) => {
    if (response.ok) return response;  // 成功直接返回

    if (response.status === 401) {
        Logout();
        return response;
    }

    if (isBrowser()) {
        const body = await response.clone().json();
        toast.error(body.detail || body.title || 'Unknown error');
    }

    throw new Error(`API error: ${response.status}`);
});
```

不再需要 `skipErrorHandler` 或 `silentClient`。对于不想 toast 的场景，在组件级别 catch error 即可：

```typescript
// 替代 skipErrorHandler
try {
    const { data } = await getGlobalConfig();
} catch {
    // 静默处理
}
```

### 2.8 OpenAPI spec 导出

```go
// cmd/openapi.go — 新增 CLI 子命令
func exportOpenAPI(cmd *cobra.Command, args []string) {
    // 注意：只注册路由元数据，不需要完整的 ServiceContext
    // RegisterHandlersForSpec() 传 nil svcCtx，huma 只读取类型信息生成 schema
    // handler 函数体不会被调用，只需要 Input/Output 类型签名
    router := gin.New()
    api := handler.RegisterHandlersForSpec(router)

    // 序列化 OpenAPI spec
    b, _ := json.MarshalIndent(api.OpenAPI(), "", "  ")
    os.WriteFile("docs/openapi/admin.json", b, 0644)
}

// internal/handler/routes.go
// RegisterHandlersForSpec 只注册路由元数据（用于 CI 导出 spec）
// handler 函数传 nil closure，不需要真实的 ServiceContext
func RegisterHandlersForSpec(router *gin.Engine) huma.API {
    api := humagin.New(router, huma.DefaultConfig("Perfect Panel API", "1.0.0"))
    // huma.Register 只需要 Operation 元数据 + Input/Output 类型
    // handler 函数可以是空实现（spec 导出时不会被调用）
    huma.Register(api, huma.Operation{...}, func(ctx context.Context, input *CreateAdsInput) (*CreateAdsOutput, error) {
        return nil, nil // spec-only stub
    })
    return api
}
```

### 3.5 API.* 兼容层

`apps/admin/services/_compat/api-namespace.ts` 将 hey-api 生成的具名导出类型桥接到现有的 `declare namespace API { }` 全局命名空间。现有 79 个文件（admin）和 21 个文件（user）使用 `API.Ads`、`API.CreateAdsRequest` 等全局类型。兼容层自动从 `types.gen.ts` 提取所有 `export type` 声明，生成 `declare global { namespace API { type X = AdminTypes.X; } }` 映射。渐进迁移完成后（所有全局 API.* 改为具名 import），删除此文件。

## Open Questions

### P0（实施前验证）

1. **huma + Gin middleware 数据传递** — 现有 JWT middleware 通过 `c.Set("userId", ...)` 存储用户信息。huma handler 通过 `context.Context` 获取。需要验证 `c.Request.Context()` 到 huma 的 `ctx` 参数传递链路是否完整。
2. **types.go 的 validate tag 兼容性** — 现有 types.go 使用 `validate:"required"` tag（go-playground/validator 格式）。huma 内置验证使用不同的 tag 格式（`minLength:"1"` 等）。需要确认是否可以通过 huma 的 Resolver 接口桥接，或需要逐步迁移 tag 格式。

### P1（实施中确认）

3. **多 API 分组的 huma 实例管理** — admin/common/user/server 四组 API 有不同的中间件。**默认方案：多个 huma API 实例**（admin、common、user 各一个，server 组不需要 OpenAPI），分别产生 `admin.json`、`common.json`、`user.json`，与前端的 services/admin、services/common 目录对齐。spike 中验证多实例是否能挂在同一个 Gin Engine 上。
4. **渐进式迁移的路由冲突** — 迁移期间同一 path 能否同时存在 Gin 和 huma 注册？需要 spike 验证。

## What I noticed about how you think

- 你从 "goctl-openapi 太小了" 出发，直接跳到了 "为什么只在 go-zero 生态里找？" 这个问题问得好。不满足于 local optimum，去找 global optimum。
- 当发现项目实际用的是 Gin 而不是 go-zero 时，你没有纠结沉没成本（之前设计文档的工作），而是直接说 "那就抛弃 .api 文件"。对方向的判断果断。
- 选择 "切换到正确 HTTP 状态码" 而不是 "保持兼容" — 这意味着更大的迁移范围，但你选了更干净的架构。这是工程判断力。
